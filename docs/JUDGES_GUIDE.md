## üìç Evidence 1: Meaningful CRE Usage

Cognivern uses **five CRE capabilities** in a single forecasting workflow:

| CRE Capability | What It Does in Cognivern |
| :--- | :--- |
| **Cron Trigger** | Schedules the forecasting pipeline every 10 minutes |
| **HTTP Fetch** | Fetches Sapience prediction market data with DON consensus |
| **Confidential HTTP** | Calls LLM APIs privately ‚Äî reasoning stays in enclave until attested |
| **EVM Read** | Reads Chainlink Data Feeds (price data) as forecasting inputs |
| **EVM Write** | Submits EAS attestations to Arbitrum with multi-node verification |

**Why CRE matters here**: A prediction market agent that fetches data from a single endpoint and submits via a single RPC is vulnerable to manipulation. CRE ensures every step is independently verified by multiple DON nodes.

## üõ†Ô∏è Technical Differentiators

### 1. End-to-End Decentralization via CRE
Every step of the prediction pipeline ‚Äî data fetching, price reading, LLM reasoning, attestation submission ‚Äî runs across a DON with BFT consensus. No single point of failure.

### 2. Chainlink Data Feeds as Forecasting Inputs
The agent reads ETH/USD and BTC/USD price feeds via EVM Read capability, using real-time on-chain price data to inform its probability estimates.

### 3. Multi-LLM Orchestration
A custom failover layer within the CRE callback ensures 100% uptime: if Routeway.ai (Kimi K2) is throttled, the agent falls back to Groq (Llama 3.3) ‚Äî all within a single CRE execution.

### 4. Horizon-Weighted Market Selection
The agent doesn't pick random markets. It calculates time-until-resolution for each market and prioritizes those with the longest horizon, mathematically optimizing for accuracy scoring.
